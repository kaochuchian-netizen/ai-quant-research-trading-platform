# Strategy Metrics Definition

本文件定義 04 階段策略回測與策略排行榜使用的核心指標。04 階段仍維持 research-only，指標用途是協助研究、比較與風險檢查，不代表策略可直接進入正式 LINE 推播或自動交易。

## 1. 指標設計原則

策略績效不能只看勝率，也不能只看累積報酬。高勝率可能伴隨單次大虧，累積報酬也可能由少數極端交易貢獻，若忽略風險與樣本數，容易高估策略品質。

04 階段的指標設計應同時考慮以下面向：

- 報酬：策略是否具備正期望值。
- 風險：策略是否承受過大回撤或報酬波動。
- 樣本數：結果是否有足夠交易筆數支撐。
- pending：尚未完成或尚無法計算報酬的信號是否過多。
- 穩定度：績效是否集中在少數交易、少數股票或單一期間。

所有指標都必須能由 `strategy_signal_logs.csv` 或 backtest records 計算。若某項指標暫時缺少必要欄位，應先在文件中定義計算語意，再於後續 task 補齊資料欄位，不應用不可追溯或人工主觀判斷替代。

## 2. 核心指標總覽

| 指標 | 英文欄位名 | 主要用途 |
| --- | --- | --- |
| 交易筆數 | `trade_count` | 判斷可計算交易樣本數是否足夠 |
| Pending 筆數 | `pending_count` | 判斷尚未完成或無法計算結果的信號數 |
| Pending 比例 | `pending_ratio` | 判斷未完成信號是否過度影響績效判讀 |
| 勝率 | `win_rate` | 衡量可計算交易中獲利交易比例 |
| 平均報酬 | `avg_return` | 衡量單筆交易平均期望報酬 |
| 累積報酬 | `cumulative_return` | 衡量策略期間整體複利報酬 |
| 最大回撤 | `max_drawdown` | 衡量策略資金曲線最大下跌風險 |
| 類 Sharpe 分數 | `sharpe_like_score` | 衡量報酬相對波動的風險調整表現 |
| 穩定度分數 | `stability_score` | 衡量績效在交易、股票或期間上的穩定程度 |

## 3. 指標定義

### 3.1 交易筆數

- 中文名稱：交易筆數
- 英文欄位名：`trade_count`
- 計算邏輯：在指定策略、持有天期與回測期間內，已完成且具備可計算報酬的交易筆數。通常等於 `return_pct`、`future_return_pct` 或對應持有天期報酬欄位不為空的筆數。
- 用途：判斷策略樣本是否足夠，避免少數交易造成排名失真。
- 風險或解讀限制：`trade_count` 高不代表策略好，只代表樣本較多；仍需搭配報酬、回撤與穩定度判斷。

### 3.2 Pending 筆數

- 中文名稱：Pending 筆數
- 英文欄位名：`pending_count`
- 計算邏輯：在指定策略、持有天期與回測期間內，尚未能計算最終報酬的信號筆數。常見原因包含未來價格資料尚不足、持有天期尚未結束、資料缺漏或回測資格不足。
- 用途：衡量策略結果尚未完成的程度，避免排行榜把未完成信號視為完整績效。
- 風險或解讀限制：pending 不一定代表壞交易，但 pending 過多時，已完成交易可能無法代表完整策略結果。

### 3.3 Pending 比例

- 中文名稱：Pending 比例
- 英文欄位名：`pending_ratio`
- 計算邏輯：`pending_count / (trade_count + pending_count)`。若分母為 0，結果應標記為無法評估，而不是視為 0。
- 用途：判斷策略結果完成度。排行榜應限制 pending 比例過高的策略進入正式比較。
- 風險或解讀限制：近期信號較多的策略可能自然有較高 pending 比例，因此應搭配回測期間與持有天期解讀。

### 3.4 勝率

- 中文名稱：勝率
- 英文欄位名：`win_rate`
- 計算邏輯：`獲利交易筆數 / trade_count`。獲利交易通常定義為已完成交易報酬大於 0。呈現時可使用百分比。
- 用途：衡量交易方向命中率，協助比較策略訊號品質。
- 風險或解讀限制：勝率不可單獨作為策略好壞依據。高勝率若平均獲利小、平均虧損大，仍可能是負期望值。

### 3.5 平均報酬

- 中文名稱：平均報酬
- 英文欄位名：`avg_return`
- 計算邏輯：已完成交易報酬的算術平均，公式為 `sum(returns) / trade_count`。報酬單位需與 backtest records 保持一致，初期可使用百分比。
- 用途：衡量單筆交易的平均期望值，是策略比較的核心報酬指標之一。
- 風險或解讀限制：平均值容易受極端值影響。若樣本數低或報酬分布高度偏斜，應搭配 cumulative return、max drawdown 與 stability score 解讀。

### 3.6 累積報酬

- 中文名稱：累積報酬
- 英文欄位名：`cumulative_return`
- 計算邏輯：依交易報酬進行複利累乘，公式為 `product(1 + return_pct / 100) - 1`，呈現時可轉為百分比。若後續定義資金曲線與倉位配置，應以資金曲線口徑為準。
- 用途：衡量策略在回測期間的整體報酬表現。
- 風險或解讀限制：累積報酬可能被少數極端獲利交易主導，也可能未反映資金使用率、交易重疊、交易成本與滑價。

### 3.7 最大回撤

- 中文名稱：最大回撤
- 英文欄位名：`max_drawdown`
- 計算邏輯：先依策略交易順序建立累積資金曲線，再計算任一時間點相對歷史高點的最大下跌幅度。公式可表示為 `min((equity - peak_equity) / peak_equity)`，呈現時通常使用負百分比或正的回撤幅度。
- 用途：衡量策略最差資金曲線下跌風險，是排行榜與正式啟用前的重要風控指標。
- 風險或解讀限制：若交易順序、持倉重疊、倉位大小或交易成本尚未完整建模，初期 max drawdown 只能視為近似風險指標。

### 3.8 類 Sharpe 分數

- 中文名稱：類 Sharpe 分數
- 英文欄位名：`sharpe_like_score`
- 計算邏輯：以已完成交易報酬的平均值除以報酬標準差，公式為 `avg_return / std_return`。若 `trade_count < 2` 或標準差為 0，應標記為無法評估。
- 用途：衡量策略報酬相對波動的風險調整表現，避免只看平均或累積報酬。
- 風險或解讀限制：這是簡化版 Sharpe，不一定年化，也未必扣除無風險利率。交易報酬若非獨立分布或樣本數太低，分數可信度有限。

### 3.9 穩定度分數

- 中文名稱：穩定度分數
- 英文欄位名：`stability_score`
- 計算邏輯：初期可由多個穩定性檢查組合而成，例如分期間平均報酬一致性、分股票績效集中度、獲利交易占累積報酬貢獻是否過度集中、以及 `sharpe_like_score` 是否維持合理。若資料尚不足，可先以可計算子項目建立 0 到 100 的輔助分數。
- 用途：判斷策略績效是否穩定，避免策略只靠單一股票、單一期間或少數交易撐高排名。
- 風險或解讀限制：穩定度分數需要明確版本化。不同公式不可混用，排行榜應顯示計算版本或在文件中固定語意。

## 4. 排行榜建議使用方式

策略排行榜不應使用單一指標排序。初期建議先用門檻排除不適合正式比較的策略，再用多指標綜合排序。

建議流程如下：

1. `trade_count` 需達最低門檻，樣本太少的策略只列為觀察，不納入正式排名。
2. `pending_ratio` 不可過高，避免未完成信號扭曲已完成績效。
3. `max_drawdown` 不可過大，需先通過基本風控檢查。
4. 通過門檻後，再用 `avg_return`、`cumulative_return`、`sharpe_like_score` 綜合排序。
5. `stability_score` 作為輔助指標，用於識別績效是否集中或不穩定。

初期排行榜可把策略分為以下狀態：

- `research_only`：僅供研究，不可作為正式推播依據。
- `insufficient_sample`：`trade_count` 不足。
- `too_many_pending`：`pending_ratio` 過高。
- `risk_rejected`：`max_drawdown` 超過可接受範圍。
- `candidate_watchlist`：通過初步門檻，但仍需更多期間外驗證與人工審核。

## 5. 初期最低門檻建議

初期門檻應保守，且所有策略結果先維持 research-only。以下為 04 階段早期建議，不代表正式啟用條件：

- `trade_count` 太低不可正式評估。初期可先要求至少 20 筆已完成交易；低於門檻者只能列為觀察。
- `pending_ratio` 太高不可正式判斷。初期可先要求不高於 30%；超過門檻者應標記為結果未成熟。
- `max_drawdown` 必須納入風控。初期不應讓高報酬但大回撤策略直接排在前面，具體上限需依策略持有天期與風險承受度另行定義。
- `sharpe_like_score` 必須搭配樣本數使用。樣本數不足時，即使分數很高也不應視為穩定策略。
- `stability_score` 初期只作輔助，不應取代報酬與風控指標。
- 策略結果先維持 research-only，不直接影響正式 LINE 推播、不自動下單、不改變 production 策略。

## 6. 後續實作注意事項

後續若要將本文件落實到程式，應優先保持欄位命名一致，並確保每個指標都能從 `strategy_signal_logs.csv` 或 backtest records 追溯計算。新增欄位時應同步更新回測輸出、排行榜輸出與相關文件，避免同一指標在不同模組有不同語意。
