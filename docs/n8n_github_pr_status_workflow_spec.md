# n8n GitHub PR Status Workflow Spec

## Purpose

This document defines how n8n participates in AI-DEV by collecting GitHub pull
request status, CI status, changed files, mergeable state, and producing a PR
status report.

The workflow is documentation-only and does not create a formal n8n workflow,
configure GitHub credentials, or authorize any merge or production action.

## Background

The AI Dev closed loop already uses GitHub PRs, GitHub Actions, local
validation, and post-merge closeout. This spec adds a narrow status-collection
role for n8n so the automation layer can summarize what is already happening
without replacing the human merge gate.

AI-DEV-031 is the design step for that reporting flow. It does not create a
runtime workflow.

## Workflow Goal

The goal is to produce a consistent PR status report that answers the following
questions:

- Is the PR open?
- Is the PR mergeable?
- What is the merge state?
- Which checks passed or failed?
- Which files changed?
- Is the PR ready for human merge review?

The output should be a report that can feed dashboards, merge reviews, or
closeout summaries, but it must not auto-merge anything.

## n8n Role in AI-DEV

n8n acts as the status and coordination layer.

In this workflow, n8n may:

- fetch GitHub PR metadata
- collect GitHub Actions status
- summarize changed files
- assemble a PR status report
- pass the report to Dify, ChatGPT, or a human reviewer
- track progress metadata for the AI-DEV loop

n8n must not:

- configure or store GitHub credentials in this spec task
- create a formal workflow definition
- mutate the repository
- approve its own merge result
- send notifications without a separate approved task

## Workflow Input

The workflow input should be a compact PR reference:

- repository owner and name
- PR number
- branch name
- base branch
- optional human context or task_id

For AI-DEV usage, the input is expected to come from GitHub PR metadata or a
task package generated by Codex or Dify.

## GitHub Data to Collect

n8n should collect the following GitHub data points:

- PR number
- PR URL
- PR title
- PR state
- head branch
- base branch
- mergeable flag
- merge state status
- changed files list
- changed file count
- GitHub Actions workflow names
- GitHub Actions run status
- GitHub Actions conclusion
- check suite or required check names
- check suite or required check conclusions

If additional details are needed, they should be clearly labeled as optional
and should not be required for the core report contract.

## Output Report Contract

The PR status report should be a structured JSON document that can be rendered
by other tools.

The contract should at least include:

- task_id
- pr_number
- pr_url
- branch
- base_branch
- state
- mergeable
- merge_state_status
- checks.required_check
- checks.status
- files_changed
- recommendation
- manual_gate_required
- risk_flags.production_pipeline_changed
- risk_flags.db_changed
- risk_flags.cron_or_timer_changed
- risk_flags.secrets_or_env_changed
- risk_flags.notification_changed
- risk_flags.trading_or_order_logic_changed
- generated_at

The report should separate raw status values from the final recommendation so
that human reviewers can see the evidence behind the summary.

## Safety Boundary

The workflow is read-only by default.

It must not:

- modify production pipeline code
- write to the formal database
- read or modify secrets, credentials, or `.env`
- send LINE or email notifications
- execute trading or order logic
- run production-approved runners
- create or modify cron, systemd, or timer jobs
- create or mutate Dify or n8n runtime settings

## Human Gate Rules

The report may support human review, but it does not replace it.

Human approval is required for:

- PR merge
- branch cleanup
- archive completed_by_pr
- any repository mutation
- any production deployment proposal
- any notification workflow proposal
- any dashboard deployment proposal

If the report indicates a problem, the correct response is human review and
follow-up work, not automatic mutation.

## Example PR Status Report

An example report should look like this:

```json
{
  "schema_version": 1,
  "task_id": "AI-DEV-031",
  "pr_number": 31,
  "pr_url": "https://github.com/example-org/example-repo/pull/31",
  "branch": "ai-dev/example",
  "base_branch": "main",
  "state": "OPEN",
  "mergeable": true,
  "merge_state_status": "CLEAN",
  "checks": {
    "required_check": "AI Dev Validation",
    "status": "success"
  },
  "files_changed": [
    "docs/example.md"
  ],
  "risk_flags": {
    "production_pipeline_changed": false,
    "db_changed": false,
    "cron_or_timer_changed": false,
    "secrets_or_env_changed": false,
    "notification_changed": false,
    "trading_or_order_logic_changed": false
  },
  "recommendation": "ready_for_human_merge_review",
  "manual_gate_required": true,
  "generated_at": "2026-06-24T08:00:00+08:00",
  "notes": [
    "Example PR status report for n8n GitHub integration planning only.",
    "This example does not authorize merge, archive, notifications, or trading."
  ]
}
```

This example is a reporting example only. It is not a workflow execution
contract.

## Future Dify Integration

Future Dify integration should treat this PR status report as a reusable input
artifact for:

- task package generation
- roadmap reconciliation
- validation summary drafting
- closeout summary drafting
- human merge-gate review support

Dify should not own GitHub credentials for this task, and it should not turn
the report into an automatic merge instruction.

## Acceptance Criteria

AI-DEV-031 is complete when:

- this spec document exists
- the workflow goal is limited to PR status reporting
- the GitHub data collection list is explicit
- the output report contract is explicit
- the safety boundary excludes secrets, production, DB, cron, trading, and
  runner usage
- human gate rules require approval for merge, archive, and cleanup
- an example PR status report is included
- future Dify integration is described as downstream and read-only at this
  stage
- no formal n8n workflow or GitHub credential configuration is created
